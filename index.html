¿Qué es Web Scraping?

Extracción automatizada de datos de sitios web

Casos de uso:

  • Monitoreo de precios y competencia
  • Investigación académica y análisis de datos
  • Agregación de contenido (noticias, productos)
  • Búsqueda de empleo y oportunidades
  • Análisis de mercado y tendencias

¿Cómo funciona la web?

El navegador recibe tres tipos principales de archivos:

HTML

Estructura el contenido

CSS

Define el estilo visual

JavaScript

Añade interactividad

También: imágenes, fuentes, datos (JSON), etc.

Dos significados de "Rendering"

Rendering a bajo nivel

Lo que hace el navegador:

  • HTML → DOM tree
  • CSS → CSSOM
  • Render tree → Layout
  • Paint → Píxeles en pantalla

Siempre ocurre en el navegador

Rendering arquitectural

Lo que nos importa para scraping:

¿Quién construye el HTML/DOM?

  • ¿El servidor?
  • ¿El navegador (JavaScript)?

Server-Side Rendering (SSR) y Static Sites

SSR

  • El servidor ejecuta código por cada request
  • Genera HTML completo con los datos
  • Ejemplo: Django, PHP, Next.js SSR

Static / SSG

  • HTML pre-generado en build time
  • Servidor solo sirve archivos
  • Ejemplo: Hugo, Gatsby, Jekyll
Para scraping: Ambos lucen igual → HTML ya contiene los datos

Client-Side Rendering (CSR)

¿Cómo funciona?

  • Servidor envía HTML "shell" mínimo + bundles de JS
  • JavaScript en el navegador hace fetch de datos (APIs)
  • JavaScript construye y actualiza el DOM
  • HTML inicial está prácticamente vacío

Ejemplo: React SPA clásico, Vue SPA, Angular SPA

<!-- HTML inicial típico de CSR -->
<div id="root"></div>
<script src="bundle.js"></script>

La pregunta clave para scraping

"¿El HTML que descargo ya contiene los datos que necesito?"

✓ SÍ

SSR / SSG

El HTML está completo desde el servidor

✗ NO

CSR

Solo hay un shell + JavaScript

Esto determina tu estrategia de scraping

Cómo identificar SSR/SSG vs CSR

Tres tests rápidos:

1. View Source vs Inspect

• View Source = HTML del servidor

• Inspect = DOM después de JS

¿Datos en View Source? → SSR/SSG

2. Curl / requests

• curl https://sitio.com

¿Respuesta tiene contenido? → SSR/SSG

¿Solo <div id="root">? → CSR

3. Desactivar JavaScript

• Desactivar JS en el navegador

• Recargar la página

¿Visible? → SSR/SSG

¿En blanco? → CSR

Tres estrategias de scraping

1. HTML + DOM Parsing

Cuándo: HTML ya tiene los datos

Herramientas: requests + BeautifulSoup, lxml, Cheerio

2. API Scraping

Cuándo: Página usa CSR + AJAX/fetch

Método: DevTools → Network tab → llamar APIs directamente

3. Headless Browser

Cuándo: JS complejo, auth, anti-bot

Herramientas: Selenium, Playwright, Puppeteer

Estrategia 1: HTML Parsing

Cuándo usarla:

  • View Source muestra los datos
  • SSR, SSG, o páginas tradicionales

Proceso:

  1. Hacer HTTP request (requests.get())
  2. Parsear HTML (BeautifulSoup/lxml)
  3. Extraer elementos con selectores CSS o XPath
import requests
from bs4 import BeautifulSoup

response = requests.get('https://ejemplo.com')
soup = BeautifulSoup(response.text, 'html.parser')
productos = soup.find_all('div', class_='producto')

Estrategia 2: API Scraping

Cuándo usarla:

  • Página usa CSR
  • DevTools muestra llamadas XHR/fetch con JSON

Proceso:

  1. Abrir DevTools → Network tab
  2. Identificar endpoints que retornan datos
  3. Replicar esos requests en tu código
  4. Parsear JSON directamente

Ventajas:

  • Más limpio que parsear HTML
  • Más rápido (no necesitas el DOM)
  • Datos ya estructurados

Estrategia 3: Headless Browser

Cuándo usarla:

  • No encuentras las APIs fácilmente
  • Página tiene anti-bot, CAPTCHA
  • Requiere interacciones complejas (login, clicks, scroll)

Proceso:

  1. Automatizar navegador (Selenium/Playwright)
  2. Cargar página, esperar a que JS ejecute
  3. Interactuar si es necesario
  4. Leer el DOM resultante

Tradeoff: Más lento y consume más recursos, pero más robusto

Comparación de estrategias

Aspecto HTML Parsing API Scraping Headless Browser
Velocidad ⚡⚡⚡ Rápido ⚡⚡⚡ Rápido ⚡ Lento
Complejidad 🟢 Simple 🟡 Media 🔴 Alta
Casos de uso SSR/SSG CSR con APIs visibles CSR complejo, anti-bot
Recursos Bajo Bajo Alto (CPU, memoria)
Mantenimiento Medio (cambios en HTML) Bajo (APIs estables) Alto (cambios frecuentes)

Consideraciones prácticas

Ética y legalidad

  • Respetar robots.txt
  • Leer y seguir los Terms of Service (ToS)
  • Considerar aspectos legales por país/región

Aspectos técnicos

  • Rate limiting: no sobrecargar servidores
  • Manejo de paginación e infinite scroll
  • Rotación de IPs/headers para evitar bloqueos
  • Validación y manejo de errores (páginas cambian)
  • User-Agent apropiado

Ejemplo de código

Scraper simple con BeautifulSoup

import requests
from bs4 import BeautifulSoup

# 1. Obtener HTML
url = 'https://ejemplo.com/productos'
response = requests.get(url)

# 2. Parsear
soup = BeautifulSoup(response.text, 'html.parser')

# 3. Extraer datos
for producto in soup.find_all('div', class_='producto'):
    nombre = producto.find('h2', class_='titulo').text
    precio = producto.find('span', class_='precio').text
    print(f"{nombre}: {precio}")
